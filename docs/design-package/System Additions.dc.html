Design follow-up · surfaces 10–14

The surfaces we hadn't covered

Five pieces that sit around the reports: the daily digest email (your highest-frequency touchpoint), a real navigation model, a state kit for empty/error/no-data, a dark-mode render to validate the tokens, and the first-run / unmapped-queue experience. All built on the same cool palette and ds- components from the handoff.

10 · Daily digest email

The email most managers actually read

If managers glance at the email and only click in when something's off, this template drives perceived quality more than any screen. It must answer "are we OK?" in the preview pane, deep-link the one dept that needs attention, and survive email clients — so: table-based layout, all styles inline, web-safe + system fonts, no JS, ~600px. Build rules
  • Tables, not flex/grid. Outlook renders on Word's engine — lay out with nested <table>, fixed widths, role="presentation".
  • Every style inline. No <style> reliance; Gmail strips much of it. Mirror the tokens as literal hex.
  • System font stack — Inter Tight won't load in most clients. Use -apple-system, "Segoe UI", Arial; keep the display weight via font-weight:700.
  • Bars = filled table cells, not SVG/CSS gradients. The target bar is a 2px black <td>.
  • One job: "are we OK?" The hero %, the verdict line, and the deep-link must all be above the fold / in preview text.
  • Real preheader — the hidden snippet that becomes the inbox sub-line; set it deliberately.
  • Bulletproof button — padded anchor with bg color, not a styled <button>.
  • Dark-mode email — set color-scheme + a dark media block; many clients invert, so avoid pure-white-only logic.
  • Per-recipient relevance — lead with their dept; the company table is supporting context.
Why this matters → It's the same visual language as the dashboard (hero %, target marker, at-a-glance banner, status bars) so the email and app feel like one product — and it does the manager's triage for them before they ever click.

One component, five tones
  • icon tile · headline · one plain sentence · ≤2 actions

  • always offer the next step — preset, retry, or a way back
  • never discard last-good data on error — show it, flagged

The override block
/* only neutrals + soft semantics swap.
   --accent/--good/--warn/--bad hues stay. */
[data-theme="dark"] {
  --paper:#0e151c; --paper-2:#243441;
  --paper-card:#15202b;
  --ink:#e7eef5; --muted:#6b7d8f;
  --line:#243441; --line-2:#1c2935;

  /* brighten softs so chips/rails read on dark */
  --good:#46a888; --good-soft:#10322a;
  --warn:#cf7e5e; --warn-soft:#3a201a;
  --bad:#d9684f;  --bad-soft:#3a1a16;
  --accent:#5b8fd6; --accent-soft:#16273b;
}
Findings →
  • Components needed zero markup changes — proof the token approach holds.
  • The only real work is brightening the semantic softs (a dark --good-soft chip on a dark card needs more contrast).
  • Sparkline strokes use --good/--warn, so they re-color for free.
  • Pure-black is avoided; #0e151c keeps the cool cast.
14 · First-run & unmapped queues

The roughest first impression, designed deliberately

A brand-new workspace — or a newly discovered queue — produces empty reports with no explanation, which reads as "the product is broken." The fix is a guided setup checklist that turns the blank state into a clear, finishable task, and a persistent (dismissible) banner whenever queues go unmapped.
