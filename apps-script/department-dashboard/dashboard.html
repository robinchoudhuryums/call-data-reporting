Help & FAQ

Quick guide to the table columns

  • Agent · Source — every agent with data in the selected range appears, tagged in the Source column: ROSTER (on your department roster), QUEUE (a floater from another dept who handled your queue extensions — their home dept follows the chip), or BOTH. Roster totals exclude queue-only floaters; the floater subtotal is shown separately in the table footer. The Diagnostics panel lists roster agents with no data in range.
  • Unique vs Rung — “Unique” is a single caller attempt; “Rung” is the number of times an agent was notified of an incoming call. A caller on hold for 2 minutes who disconnects counts as 1 Unique but several rings. Total Missed and any “% Answered” metric are based on Rung, not Unique.
  • TTT · ATT — Total Talk Time and Average Talk Time. Only counts inbound calls from a call queue.
  • Avg Abd Wait · CSR Avg Abd Wait — Average abandoned-caller wait time, dept-wide and CSR-specific.

Source column (Roster / Queue / Both)

  • ROSTER: an agent officially assigned to your department in the roster sheet.
  • QUEUE: a floater not on your roster who handled your department’s queue extensions — useful for spotting cross-coverage. The chip names their home department(s).
  • BOTH: on your roster and matched via queue this period (the common case for active members).

There is no scope toggle anymore — the table always shows the full picture, and the Source column makes each agent’s provenance explicit. Department totals are roster-only; queue-only floaters are summed on a separate footer line so headline numbers stay stable.

Common questions

  • Why is my table empty? Check the From/To dates. If they look right, the “Roster agents with no data in this range” section of the Diagnostics panel will list anyone expected but absent.
  • Why do some numbers differ from the old DQE Report spreadsheet? The legacy report had a few bugs around the work-window (calls before 8:30 AM or after 5:00 PM CST were sometimes counted in TTT/ATT but not Answered). The dashboard reads the same source sheet and inherits whatever it currently produces. If a specific number still looks wrong after a source rebuild, email Robin with date + agent.
  • Why are there more missed-call timestamps for an agent than the official “Missed” count? Some rings happen before/after the 6:30 AM – 3:00 PM PST work window. Those appear as raw timestamps but don’t count in “Total Missed”. See the Missed Calls report (Reports section above the table) for the full timestamp list per agent.
  • What if an agent appears in multiple departments? That’s fine — agents can be on multiple rosters. The same daily numbers will show up on each department’s view that they’re rostered under.
  • Why does the manager see only their department? Access is controlled by an Email → Department mapping. Managers see only their own department; admins can switch via the dropdown.

Reports overview

All legacy DQE Report features are now migrated to this dashboard:

  • Individual Report — per-agent KPIs + 12-month trend, with peer-comparison mode for multiple agents.
  • Performance Report — dept-level current-vs-prior-period deltas, share-of-answered pie, and 12-month team trend.
  • Compare Ranges — side-by-side comparison of two arbitrary date ranges.
  • Alerts (admin-only) — per-dept low answer rate alert emails, manually or via a daily trigger.

Caller Lookup

Traces every inbound call from this number in inbound_calls (Neon). The number is normalized and HMAC-hashed before querying — it is never stored, logged, or sent anywhere raw. Calls captured since the journey extension show the full leg-by-leg path (IVR → queues → agent rings → answer / hold); older rows show the entry → final queue summary.

Daily trigger

The daily trigger runs at 8 AM and checks the previous day's answer rate per active dept. Weekends (Sat / Sun) are skipped automatically.

Neon keep-warm

Pings Neon every few minutes during a weekday business-hours window so the free-tier instance stays warm and DQE reads (when DQE_READ_SOURCE=neon) don't pay a cold-start penalty. Window defaults to 7 AM–1 PM Central; tune via the NEON_KEEPWARM_START_HOUR / NEON_KEEPWARM_END_HOUR Script Properties (watch the estimated monthly compute-hours below to stay under your Neon plan).

Report cache warming

Pre-warms the Overview blob + each dept's My Department default-range summary once a day (after the morning ingest) so the first manager to open the dashboard gets a cache hit instead of a cold aggregation. Fires at CACHE_WARM_HOUR Script Property (default 9 AM Central). "Warm now" runs it on demand.
